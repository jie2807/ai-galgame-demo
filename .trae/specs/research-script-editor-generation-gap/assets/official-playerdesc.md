## 四、运行时消费路径

SimpleChatEngine._buildSystemPrompt（[index.html](file:///d:/BC/qmzz/index.html) L11662-L11760）在 L11741/L11745 把 `chapter.playerDesc` 直接字符串拼接到 `playerSection`：

```javascript
playerSection = '【玩家角色】\n玩家扮演' + chapter.playerName + '，' + (chapter.playerTitle || '冒险者') + '。' + (chapter.playerDesc || '');
```

**关键观察**：运行时把整个 `playerDesc`（含四段式结构）作为 system prompt 的一部分发送给 AI。这意味着官方剧本的"主控文档"模式工作正常——AI 收到约 1900 字的 playerDesc，其中明确列出了 10 个状态变量、12 个时间线事件、4 个终局条件，AI 会按这些指令叙事。

而 AI 生成的自定义剧本，playerDesc 仅 50-300 字，且 normalize 后写入 `customPlayerDesc` 输入框（[index.html](file:///d:/BC/qmzz/index.html) L8699，`rows="2"` 的小 textarea），运行时被拼接进 system prompt 后完全无法承载"主控文档"职能。

## 五、主控文档模式的本质

官方 `playerDesc` 不是"角色描述"，而是**剧本主控文档**——它是一个用自然语言写就的、包含完整剧情骨架的元数据文档，通过 system prompt 注入给 AI，让 AI 在叙事时遵循：

1. **状态跟踪协议**：明确告知 AI 需要跟踪 10 个状态变量（即使代码层 stateVars 字段未在运行时被读取，AI 也会按 playerDesc 文本指令跟踪）
2. **事件触发协议**：明确列出 12 个时间线事件 + 8 个阈值事件 + 8 个随机事件 + 4 个终局事件的触发条件与选项
3. **叙事风格协议**：明确 NPC 称呼规则、信息呈现方式、语言基调
4. **多结局协议**：明确 4 个终局事件的触发条件组合（如 `unity>60 且 foodDays>30`）

这是一种"用 prompt engineering 替代代码逻辑"的设计——把剧情骨架编码进 system prompt 文本，让 AI 充当运行时引擎，而非依赖代码层的状态机。AI 生成剧本完全没有这一层，所以即使代码层修复了 normalize/validate，AI 生成的剧本依然不会有剧情骨架，除非 Prompt 模板要求 AI 输出主控文档式的 playerDesc。

## 六、改进方向

1. **Prompt 模板重写**：要求 AI 输出 1500-2000 字的 playerDesc，含四段式结构
2. **UI 扩展**：`customPlayerDesc` 输入框从 `rows="2"` 扩展为 `rows="20"` 并提供四段式模板占位符
3. **validate 扩展**：校验 playerDesc 字数 >= 800 字，且含 `【AI叙事指令】`/`【AI需跟踪状态】`/`【事件系统】` 三个标记
4. **normalize 扩展**：保留 playerDesc 原文，不截断
5. **stateVars 字段化**：把 playerDesc 中的状态变量清单同步抽取到 `stateVars` 数组字段，供未来代码层状态机消费
