# 舆论锻造所macOS桌面系统AI接入与深度优化 Spec

## Why
"活动"按钮点击后展开的舆论锻造所是一个macOS风格桌面系统，底部Dock包含6个应用软件（微信、B站、抖音、邮箱、同花顺、设置），但这些应用全部使用硬编码的静态数据，没有任何AI交互能力——微信聊天无法回复、B站评论无人响应、抖音创作面板空壳、邮箱无法智能回复、同花顺无AI分析。同时系统运行卡顿：6个应用的渲染逻辑全部内联在index.html中（约2000行）、每次打开应用都全量重建DOM、AI调用（CWE引擎的`_callAI`）无节流无缓存、流式输出逐字更新DOM导致频繁重排。需要为各应用接入AI大模型实现智能交互，并对AI Agent运行和整体性能进行深度优化。

## What Changes
- **新增**：统一AI服务层（AIServiceHub）——集中管理API配置、请求队列、缓存、错误重试、流式响应，替代CWE中分散的`_callAI`实现
- **新增**：微信AI聊天——聊天对象具备AI人格，可智能回复用户消息，支持多角色对话
- **新增**：B站AI互动——视频评论区AI回复、UP主AI私信、AI生成视频内容
- **新增**：抖音AI创作——"AI创作视频"面板接入大模型，根据主题和风格生成视频脚本和内容
- **新增**：邮箱AI助手——AI智能回复邮件、AI撰写邮件、AI摘要长邮件
- **新增**：同花顺AI分析——AI股票分析、AI资讯解读、AI投资建议
- **优化**：AI请求队列与节流——防止并发AI请求过多导致卡顿和API限流
- **优化**：AI响应缓存——相同/相似prompt的响应缓存，减少重复API调用
- **优化**：AI Agent异步执行——Agent内容生成不阻塞主线程，流式输出批量DOM更新
- **优化**：macOS桌面性能——应用窗口内容按需渲染，避免重复全量重建DOM

## Impact
- Affected specs: `restructure-cw-ai-agent-social`（Agent系统AI接入升级）、`restructure-bilibili-layout-agents`（B站AI能力增强）
- Affected code:
  - `index.html`：AIServiceHub统一服务层、6个应用的AI交互逻辑、性能优化代码（L18603-20583 macOS桌面系统区域、L20585-24000+ CWE引擎区域）
  - `pages/creator-wars/creator-wars.css`：AI交互相关UI样式

## 现状问题分析

### 问题1：6个应用全部静态硬编码，无AI交互
当前macOS桌面中6个应用的状态：

| 应用 | 当前功能 | AI能力 | 主要局限 |
|------|---------|--------|---------|
| 微信 | 聊天列表、通讯录、发现、我 | 无 | 聊天消息硬编码，发送消息后无回复；朋友圈静态 |
| B站 | 视频网格、视频详情、动态、消息、我的 | 无 | 8个视频硬编码，评论静态，搜索不工作，无法发布内容 |
| 抖音 | 竖屏滑动、评论、个人主页、AI创作面板 | 空壳 | "AI创作视频"面板有UI但"开始创作"按钮仅关闭面板，无实际AI调用 |
| 邮箱 | 收件箱、已发送、草稿、废纸篓、写邮件 | 无 | 邮件全部硬编码，回复/转发仅打开写邮件窗口，无AI辅助 |
| 同花顺 | 行情、K线、持仓、资讯、我的 | 无 | 股票数据静态，K线随机生成，资讯硬编码，无AI分析 |
| 设置 | API配置状态、重置数据 | 无 | 仅显示API配置状态和重置按钮 |

### 问题2：AI调用架构分散
当前CWE引擎有独立的`_callAI`方法（L20688-20769），与游戏对话的`SimpleChatEngine`各自实现API调用、流式解析、错误处理，无法共享缓存、队列、重试等能力。

### 问题3：AI请求无节流无缓存
- 多个应用同时发起AI请求时无并发控制
- 相同prompt重复调用API无缓存
- 无请求优先级（用户主动对话应优先于后台Agent生成）
- 无错误重试机制

### 问题4：DOM操作效率低
- 每次切换应用Tab都全量重建innerHTML（如B站切换首页/动态/消息/我的，抖音切换视频等）
- 流式AI输出逐字更新DOM导致频繁重排
- 微信聊天发送消息后全量重新渲染聊天详情

### 问题5：抖音"AI创作"空壳
抖音的"AI创作视频"面板（L19633-19653）有完整的UI（主题输入、风格选择、时长选择），但"开始创作"按钮仅调用`dyClosePublish()`关闭面板，没有任何AI生成逻辑。

## ADDED Requirements

### Requirement: 统一AI服务层（AIServiceHub）
系统 SHALL 提供统一的AI服务层，集中管理所有AI调用，替代当前CWE中分散的`_callAI`实现。

**核心能力**：
- API配置管理（统一读取/存储 apiBaseUrl、apiKey、model）
- 请求队列（最大并发数可配置，默认3）
- 请求优先级（用户交互 > 应用AI回复 > 后台Agent生成）
- 响应缓存（基于prompt哈希的LRU缓存，默认50条，5分钟时效）
- 错误重试（指数退避，最多3次）
- 流式响应统一处理
- 请求取消（AbortController统一管理）

**接口设计**：
```javascript
AIServiceHub.request({
    messages: [...],
    priority: 'high' | 'normal' | 'low',
    stream: true | false,
    onChunk: function(chunk, fullContent) {},
    onComplete: function(fullContent) {},
    onError: function(error) {},
    cacheKey: 'optional-cache-key',
    maxTokens: 2000,
    temperature: 0.9
})
```

#### Scenario: 并发请求排队
- **WHEN** 微信AI回复、B站评论AI回复、同花顺AI分析同时请求
- **THEN** 按优先级排队，最多3个请求并发执行
- **AND** 用户主动对话请求始终优先

#### Scenario: 缓存命中
- **WHEN** 相同prompt的请求在5分钟内再次发起
- **THEN** 直接返回缓存结果，不发起API调用

#### Scenario: 错误重试
- **WHEN** API请求因网络波动失败
- **THEN** 自动重试，间隔1s、2s、4s（指数退避）
- **AND** 3次重试后仍失败则回调onError

### Requirement: 微信AI聊天
微信应用 SHALL 接入AI大模型，使聊天对象具备AI人格，可智能回复用户消息。

**AI交互场景**：
1. **AI聊天回复**：用户发送消息后，聊天对象通过AI生成回复，回复风格符合该角色设定
2. **多角色人格**：每个聊天对象有不同的AI人格（张三-工作伙伴、李四-设计师、产品讨论组-多人讨论等）
3. **朋友圈AI动态**：朋友圈内容由AI生成，好友动态可AI评论
4. **AI群聊**：群聊中多个AI角色同时参与讨论

**微信AI人格设计**：
```javascript
var wechatPersonas = {
    1: { name: '张三', role: '项目经理', personality: '严谨、务实、注重效率', style: '工作沟通风格，简洁直接' },
    2: { name: '李四', role: '设计师', personality: '创意、随性、追求美感', style: '轻松幽默，偶尔发表情包' },
    3: { name: '产品讨论组', role: '群聊', personality: '多人讨论，不同观点碰撞', style: '群聊风格，多人轮流发言' },
    // ...
};
```

**Prompt策略**：
- System prompt包含角色人设和对话风格
- 对话历史保留最近10条消息
- 回复延迟0.5-2秒（模拟真人打字）
- 显示"对方正在输入..."状态

#### Scenario: 微信AI聊天
- **WHEN** 用户在微信中向张三发送消息"明天会议几点？"
- **THEN** 显示"对方正在输入..."，0.5-2秒后AI生成符合张三人设的回复
- **AND** 回复以流式打字机效果显示

#### Scenario: 微信群聊AI
- **WHEN** 用户在产品讨论组发送消息
- **THEN** 群内多个AI角色依次回复，每人风格不同
- **AND** 消息间隔1-3秒，模拟真人讨论节奏

### Requirement: B站AI互动
B站应用 SHALL 接入AI大模型，实现评论区AI回复、UP主AI私信、AI生成视频内容。

**AI交互场景**：
1. **评论区AI回复**：用户发表评论后，UP主或其他用户通过AI生成回复
2. **UP主AI私信**：用户向UP主发送私信，UP主通过AI回复
3. **AI生成视频**：用户可输入主题，AI生成视频标题、简介、关键帧文字内容
4. **AI动态**：动态页内容由AI生成，用户可评论互动

**B站AI人格设计**：
```javascript
var biliPersonas = {
    '游戏老司机': { personality: '热情、专业、喜欢深度评测', style: '游戏圈黑话+专业分析' },

    '科技前沿': { personality: '理性、前瞻、数据驱动', style: '科技圈术语+逻辑分析' },
    // ...

```

#### Scenario: B站评论区AI回复
- **WHEN** 用户在视频下发表评论
- **THEN** 1-3秒后UP主通过AI生成回复
- **AND** 回复风格符合UP主人设

#### Scenario: B站AI生成视频
- **WHEN** 用户输入视频主题"AI时代的职业选择"
- **THEN** AI生成视频标题、简介、3-5个关键帧内容
- **AND** 生成过程显示进度条

### Requirement: 抖音AI创作
抖音应用的"AI创作视频"面板 SHALL 接入AI大模型，实现真正的AI视频内容生成。

**AI交互场景**：
1. **AI视频脚本生成**：用户输入主题+选择风格+选择时长，AI生成完整的视频脚本
2. **AI评论回复**：用户发表评论后，视频作者通过AI回复
3. **AI视频推荐**：基于用户交互行为，AI生成个性化推荐理由

**抖音AI创作流程**：
1. 用户输入主题（如"城市夜景"）
2. 选择风格（科普/故事/教程/搞笑）
3. 选择时长（15秒/30秒/60秒）
4. 点击"开始创作"
5. AI生成：视频标题 → 视频描述 → 关键帧文字（3-5帧）→ 标签
6. 生成完成后，新视频添加到视频列表中

#### Scenario: 抖音AI创作视频
- **WHEN** 用户在"AI创作视频"面板输入主题"城市夜景"，选择"故事"风格，30秒时长，点击"开始创作"
- **THEN** 显示生成进度条，AI依次生成标题、描述、关键帧
- **AND** 生成完成后新视频出现在视频列表中，可滑动查看

#### Scenario: 抖音评论AI回复
- **WHEN** 用户在评论区发送评论
- **THEN** 视频作者通过AI生成回复，延迟1-2秒

### Requirement: 邮箱AI助手
邮箱应用 SHALL 接入AI大模型，实现智能邮件处理。

**AI交互场景**：
1. **AI智能回复**：查看邮件时，AI根据邮件内容生成回复建议，用户可选择使用或修改
2. **AI撰写邮件**：用户输入要点，AI生成完整邮件
3. **AI邮件摘要**：长邮件自动生成摘要，显示在邮件列表中
4. **AI邮件分类**：AI根据邮件内容自动分类和标记优先级

**邮箱AI Prompt策略**：
- 智能回复：分析邮件内容 + 发件人关系 + 上下文，生成3种回复建议（正式/简洁/友好）
- 撰写邮件：用户输入要点关键词，AI扩展为完整邮件
- 摘要：提取邮件核心信息，3-5句话概括

#### Scenario: AI智能回复邮件
- **WHEN** 用户查看张伟的会议安排邮件，点击"AI回复"按钮
- **THEN** AI生成3种回复建议（正式/简洁/友好），用户可选择一种直接发送或修改后发送

#### Scenario: AI撰写邮件
- **WHEN** 用户在写邮件窗口点击"AI撰写"，输入要点"确认参加周三会议，需要投影仪"
- **THEN** AI生成完整邮件内容，用户可编辑后发送

### Requirement: 同花顺AI分析
同花顺应用 SHALL 接入AI大模型，实现智能股票分析。

**AI交互场景**：
1. **AI个股分析**：点击股票后，AI生成该股票的分析报告（基本面+技术面+建议）
2. **AI资讯解读**：财经新闻AI解读，提取对市场的影响分析
3. **AI投资建议**：基于持仓情况，AI给出投资建议
4. **AI问股**：用户可输入问题（如"茅台还值得买吗"），AI回答

**同花顺AI Prompt策略**：
- 个股分析：基于股票名称+行业+涨跌数据，AI生成分析报告
- 资讯解读：基于新闻标题+内容，AI分析对相关股票/板块的影响
- 投资建议：基于持仓数据+盈亏情况，AI给出建议（附带风险提示）

#### Scenario: AI个股分析
- **WHEN** 用户在股票详情页点击"AI分析"按钮
- **THEN** AI生成该股票的分析报告，包含基本面分析、技术面分析、操作建议
- **AND** 报告以卡片形式展示，可展开查看详情

#### Scenario: AI问股
- **WHEN** 用户在搜索框输入"茅台还值得买吗"
- **THEN** AI基于当前数据生成回答，包含分析逻辑和建议
- **AND** 回答附带"以上分析仅供参考，不构成投资建议"免责声明

### Requirement: AI请求队列与节流
系统 SHALL 实现AI请求队列和节流机制，防止并发请求过多导致卡顿和API限流。

**队列策略**：
- 最大并发数：3（可配置）
- 优先级：用户交互(high) > 应用AI回复(normal) > 后台Agent生成(low)
- 同一应用最多1个pending请求（后续请求排队）
- 请求间隔最小500ms（防止API限流）

**节流策略**：
- 微信AI回复：对话间隔1-3秒（模拟打字）
- B站评论回复：每30秒最多1次
- 抖音AI创作：每次创作间隔10秒
- 邮箱AI回复：每次间隔2秒
- 同花顺AI分析：每次间隔5秒
- 用户主动对话：无限制（优先级最高）

#### Scenario: 多应用同时AI请求
- **WHEN** 用户同时在微信聊天、B站评论、同花顺问股
- **THEN** 请求按优先级排队，最多3个并发
- **AND** 用户当前操作的应用请求优先执行

### Requirement: AI响应缓存
系统 SHALL 实现AI响应缓存，减少重复API调用。

**缓存策略**：
- 缓存键：messages数组的JSON哈希
- 缓存容量：LRU，默认50条
- 缓存时效：5分钟（可配置）
- 缓存命中：直接返回，不发起API调用
- 缓存失效：API配置变更时清空全部缓存

**特殊缓存**：
- 同花顺个股分析：按股票代码缓存，10分钟时效
- B站视频内容：按主题+风格缓存
- 抖音AI创作：按主题+风格+时长缓存

#### Scenario: 缓存命中加速
- **WHEN** 用户短时间内重复查看同一股票的AI分析
- **THEN** 第二次直接返回缓存结果，响应时间毫秒级

### Requirement: AI Agent异步执行与流式输出优化
AI Agent的内容生成 SHALL 异步执行，流式输出使用批量DOM更新，不阻塞主线程。

**实现策略**：
- AI请求使用fetch异步执行，不阻塞UI
- 流式响应DOM更新使用`requestAnimationFrame`批量处理（每100ms合并一次）
- 打字机效果使用`textContent`而非`innerHTML`
- "正在输入..."状态使用CSS动画，不触发重排

**性能目标**：
- AI处理期间macOS桌面动画流畅（窗口拖拽、Dock动画不卡顿）
- 流式输出不造成页面卡顿
- 用户输入响应延迟不超过100ms

#### Scenario: 流式输出不卡顿
- **WHEN** 微信AI流式回复长消息
- **THEN** DOM更新以100ms为批次，打字机效果平滑
- **AND** 用户可同时拖拽窗口、切换应用

---

### Requirement: macOS桌面性能优化
macOS桌面系统 SHALL 优化应用窗口的渲染性能，避免重复全量重建DOM。

**优化策略**：
- 应用Tab切换时仅更新变化区域，不全量重建innerHTML
- 微信聊天发送消息后仅追加新消息DOM，不重新渲染整个聊天区
- B站视频详情页使用增量更新
- 抖音视频切换仅更新变化元素（背景、用户信息、操作按钮）
- 已打开的应用窗口保留DOM，切换时仅显示/隐藏

#### Scenario: 微信发送消息不重渲染
- **WHEN** 用户在微信聊天中发送消息
- **THEN** 仅追加新消息气泡到聊天区，不重新渲染整个聊天详情
- **AND** 滚动到底部，无闪烁

#### Scenario: 应用窗口切换流畅
- **WHEN** 用户在Dock中切换已打开的应用
- **THEN** 窗口直接显示/隐藏，不重新渲染内容
- **AND** 切换动画流畅，无卡顿
